# Training Methodology (Practical Overview)

## What we're doing
Training a cooperative Overcooked agent in two phases: Behavioral Cloning (BC) from
demonstrations, then PPO fine-tuning against a greedy partner. The grading metric is an
official soup-throughput score where soup count dominates (10000 points per soup).

## Phase 1: Behavioral Cloning (BC)

### Data sources
We train on three sources of demonstration data:
- **Classmates' human recordings** (~538 recordings across 58+ layouts, mixed quality).
  Weighted by tier: gold=1.0, silver=0.1, bronze=0.01. The aggressive downweighting
  keeps diverse game states without teaching bad habits from low-quality demos.
- **User's human recordings** (~40 episodes across 6 benchmark maps, 2-18 soups each).
  Gold-tier quality that shows correct coordination on hard maps.
- **Greedy vs greedy synthetic demos** (440 recordings across 22 built-in layouts).
  Generated by `train/generate_demos.py`. Both agents use `GreedyFullTaskPolicy`, which
  knows the correct soup pipeline (onion->pot->wait->dish->scoop->serve). On open maps
  this produces 6-20 soups per episode. On coordination bottleneck maps the greedy gets
  0 soups, but those recordings still teach correct navigation and item interaction.

### Why greedy demos instead of more human recordings
The greedy policy's action sequences show the exact steps needed to make soups, repeated
hundreds of times. This teaches the BC model the correct pipeline much more reliably
than human recordings, which include pauses, mistakes, and idling. On 3 of 22 layouts
greedy is useless (coordination bottlenecks) — for those we rely on user recordings.

### Architecture: GNN with topology bypass
The observation is 121-dimensional: 96 dims of featurized entity states (player positions,
held items, pot timers, etc.) plus 25 dims of static per-layout topology features.

The GNN decomposes the first 96 dims into 10 entity nodes (self, partner, onion,
tomato, dish, soup, serving, counters, pot0, pot1), processes them through a
2-layer TransformerEncoder, and produces a summary vector.

The 25 topology features are **not** passed through the Transformer. They're projected
directly and concatenated with the graph output before the actor head. This bypass
ensures layout context reaches the decision layer without being diluted by the entity
attention mechanism. The 25 features are computed once per layout from the terrain
matrix and include: grid dimensions, walkable tile count, narrow passage count,
BFS distances between key locations (player-to-pot, onion-to-pot, etc.), graph
properties (dead ends, connected components, cyclic structure, symmetry).

### Why topology features matter
The base 96-dim observation doesn't tell the model what layout it's on — all layouts
look similar at the entity level. With topology features, the model has explicit signals
like "this layout has 14 narrow passages and a central counter wall" vs "this layout
has 0 narrow passages." This lets the model learn different behavior for different
layout types without memorizing layout names. The bypass design (not through attention)
guarantees the layout signal can't be ignored by the entity attention.

### BC training details
- Optimizer: Adam, lr=1e-3, weight_decay=0
- Scheduler: ReduceLROnPlateau (halves LR when val_loss plateaus)
- Epochs: 50, batch_size=256
- Loss: tier-weighted cross-entropy (gold=1.0, silver=0.1, bronze=0.01 per-sample weight
  from filter tiers, not class weights for specific actions)
- The checkpoint stores: model state, obs_dim (121), hidden/layers (128/2), arch (gnn),
  topo_dim (25), and training history

## Phase 2: PPO Fine-Tuning

### Why PPO instead of more BC
BC can only imitate — it can't discover strategies better than the demonstrations.
On coordination bottleneck maps (counter_circuit, forced_coordination, cramped_room),
even good human recordings don't cover enough game states. PPO explores through trial
and error and gets rewarded when soups are delivered (+20 sparse reward), which lets
it discover coordination behaviors (like passing onions over a counter) that BC
can't learn from limited demos.

### Partner-aware training
The agent trains against a `GreedyFullTaskPolicy` partner on open maps. The greedy
partner is competent (delivers 6-20 soups on open maps), so it provides a useful
+20 reward signal when soups are delivered. On coordination bottlenecks where greedy
gets 0 soups (can't help), the agent switches to self-play — both agents explore
together, and one might discover coordination through exploration.

This is controlled by `--coordination-layouts` in train_ppo.py. Layouts in this list
use self-play; all others use the greedy partner.

### Preventing catastrophic forgetting
A frozen copy of the BC policy is kept in memory. A Reverse KL divergence penalty
(KL(pi_theta || pi_BC)) is added to the PPO loss. This forces the policy to stay
close to BC's action distribution — it prevents PPO from randomly exploring into
degenerate behaviors. The KL coefficient (0.3) is high enough to anchor, low enough
to allow useful exploration.

### Key PPO hyperparameters (why these values)
- lr=5e-6: Very low. Fine-tuning, not training from scratch. Higher rates cause
  divergence because the KL penalty can't fully constrain the policy.
- kl-coef=0.3: Strong enough to prevent forgetting, weak enough to allow coordination
  discovery. Earlier attempts at 0.05 caused total collapse (PPO destroyed BC within
  10k steps).
- entropy-coef=0.05: Higher than usual. On coordination layouts, the agent needs to
  try many different actions (including apparently useless ones like dropping items
  on counters) before discovering that passing over the counter leads to soup. Higher
  entropy encourages this exploration.
- timesteps=200000: Long enough for rare coordination behaviors to be discovered,
  short enough to test in reasonable time.
- rollout-len=2000: 5 episodes per batch. Enough for stable advantage estimation.
- eval-interval=20000: Check progress and save best checkpoint.
- early-stop-patience=5: Don't stop training just because of 1-2 bad evals —
  noisy exploration on coordination layouts can cause temporary dips.

### Reward shaping (plain English)
The agent gets +20 per soup delivered (the only thing that matters for the grade).
But +20 is too sparse to learn from — the agent might take hundreds of steps between
soups. The reward shaping provides dense intermediate guidance without changing the
optimal strategy.

**How the shaping works:** At every step, we compute a potential value Phi(s) for the
current state. This potential is higher when the agent is close to useful targets:
- +2.5 for holding a cooked soup (ready to deliver)
- -0.5 * distance to serving counter for soup-holders
- -0.2 * distance to nearest pot for onion/dish holders
- +1.5-4.5 per pot state improvement (onions added, cooking, ready)

The shaping reward is: gamma * Phi(next_state) - Phi(current_state). Because it's a
difference of potentials, the total shaping over any full trajectory telescopes to
gamma^T * Phi(s_T) - Phi(s_0) — a small constant that doesn't change which actions
are optimal. This guarantees the shaping guides exploration without biasing the
final policy.

An idle penalty (-0.05) applies when the agent stays still while far from its target,
discouraging hesitation.

### GAE and truncation bootstrap
Standard GAE (lambda=0.95, gamma=0.99) with one important modification: when an episode
ends due to the horizon cutoff (400 steps, not a termination), we bootstrap the value
estimate from the critic instead of zeroing it. Without this, the agent learns to give
up near step 400 (wrongly believing the episode value is 0). With it, the agent
continues trying until the last step.

The two agents' advantage estimates are computed independently (not interleaved) to
prevent advantage contamination between agents.

### Early stopping
Every 20k steps we run a quick eval (3 episodes per layout, greedy partner, argmax).
The best eval score is tracked separately. If 5 consecutive evals are below the best,
training stops and the best checkpoint is restored. This prevents wasting compute when
the policy plateaus or degrades.

## Inference

The trained agent (`TrainedAgent` in `policies/trained_agent.py`) loads the checkpoint
and a `layout_name` from the eval YAML config. At init, it computes the 25-dim topology
features for that layout. At each step, it receives a 96-dim observation from the game
engine, appends the precomputed topology features (making 121-dim), and runs the GNN
forward pass. The result is fed to the actor head, which produces action logits. With
`deterministic: true`, it takes the argmax; otherwise samples from the distribution.

Important: `PPODirectAgent` (the `trained_ppo` type) does NOT append topology features.
Only `TrainedAgent` (the `python_class` type with `layout_name` in config) does.
Using `trained_ppo` with a topology-trained model silently crashes at inference.

## What limits performance

1. **BC data quality for coordination layouts:** The greedy demos teach the pipeline
   but not coordination. User recordings show coordination but there aren't enough.
   PPO can discover coordination through exploration, but this takes time.

2. **PPO exploration budget:** Discovering counter-passing through random exploration
   requires many episodes. 200k timesteps (~500 episodes per layout) is barely enough.
   More timesteps would likely improve coordination discovery.

3. **No human demos for greedy-incompatible layouts:** On forced_coordination, the
   greedy partner gets 0 soups in self-play. Self-play PPO with two BC agents also
   struggles because BC doesn't know how to coordinate either. This layout may require
   specialized training or more human demos.
